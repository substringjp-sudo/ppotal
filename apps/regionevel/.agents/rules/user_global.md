# Global Rules (Updated)

## 응답 스타일
- 한국어로 친근하고 간결하게 답한다. 영어 용어는 그대로 둔다.
- 수정 제안할 때는 'why'를 먼저 한 줄로 설명하고 'what'을 보여준다.
- 가장 단순한 해결책을 먼저 제안한다. 과한 추상화 금지.
- 모르는 것은 추측하지 말고 물어본다.

## Plan-First 원칙
- **단순 오타, 문구 수정, 명확한 단일 에러 해결 등 단순한 사항은 계획 없이 즉시 수정한다.**
- **3개 파일 이상의 수정, 복잡한 로직 변경, 새로운 아키텍처 도입 시에는 코드를 쓰기 전에 먼저 변경 계획을 bullet으로 보여주고 승인을 기다린다.**
- 기존에 없던 라이브러리를 추가하려면 먼저 이유를 설명한다.

## 보안 / 안전
- API 키, 토큰, 시크릿을 코드에 하드코딩하지 않는다. 항상 .env 또는 secret manager 사용.
- 프로덕션 DB나 배포 환경에 영향 가는 명령은 실행 전 반드시 확인을 받는다.
- 파일 삭제, git force push, DB 마이그레이션은 명시적 승인 없이 절대 실행하지 않는다.

## 코드 스타일 (언어 무관)
- 함수 단위로 작게. 한 함수가 한 화면을 넘어가면 분리.
- 의미 있는 이름. 약어는 일반적인 것만(id, url, api).
- 주석은 'why'에만. 'what'은 코드로 표현.

## Git
- Conventional commits (feat:, fix:, refactor:, docs:, chore:).
- 커밋 메시지 한국어 본문 + 영어 prefix OK.
- 커밋 전에 lint와 typecheck를 돌린다.

## 스택 기본값 (오버라이드 없으면)
- TypeScript over JavaScript
- pnpm over npm/yarn
- 함수형 컴포넌트 + Hooks
- 상태관리는 Zustand 또는 React Query 우선 검토

## 안정성 및 에러 방지 (Stability)
- 작업을 작은 단위로 쪼개서 진행한다 (한 번에 많은 파일 수정 지양).
- 대용량 파일(JSON 등)을 읽거나 수정할 때는 필요한 부분만 다루어 컨텍스트 부하를 줄인다.
- **메모리 부족 예방**: 수천 줄 이상의 로그를 출력하는 명령어는 지양하고, `head`, `tail`, `grep` 등을 사용하여 출력을 제한한다.
- **파일 크기 관리**: 파일 하나가 너무 커지면 AI의 읽기/쓰기 오류가 잦아지므로, 컴포넌트와 함수를 적극적으로 분리하여 개별 파일 크기를 관리한다.
- 작업 중간에 수시로 `git commit`을 활용하여 상태를 안정화하고 에러 발생 시 복구 지점을 만든다.
- "Agent terminated" 에러가 발생하면 현재까지의 작업을 커밋하고 세션을 다시 시작하거나 단위를 더 쪼개어 시도한다.
